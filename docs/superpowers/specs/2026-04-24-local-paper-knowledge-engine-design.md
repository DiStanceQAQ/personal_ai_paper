# 本地论文知识引擎功能设计

## 1. 产品定位

本产品是一个本地优先的论文管理与文献上下文服务。它以 research idea 为中心组织论文，将每个 idea 作为独立空间，负责解析、索引、结构化和检索论文内容，并通过本地接口为外部 agent 提供可靠的文献上下文。

它不是聊天工具，也不是科研 agent。外部 agent 可以是 Codex、Claude Code、Cursor、ChatGPT Desktop、NotebookLM 或其他用户习惯的平台。本产品只负责把已有论文处理成 agent 更容易调用、引用和推理的知识基础。

核心目标：

- 让用户围绕一个 idea 管理相关论文。
- 保证不同 idea 的论文、索引、笔记和上下文相互隔离。
- 将论文内容转化为可检索、可追溯、可供 agent 调用的结构化知识。
- 在 agent 遇到科研问题、分析实验结果、设计后续实验时，提供文献证据和相关方法。
- 所有数据默认保存在本地，系统简单、可靠、可长期维护。

## 2. 非目标

第一版明确不做以下能力：

- 不运行或托管科研 agent。
- 不直接执行实验。
- 不管理实验数据生命周期。
- 不做云同步、团队协作或账号系统。
- 不做社交、推荐流或公开论文平台。
- 不做大而全的自动综述生成器。
- 不强绑定某一个科研领域。
- 不要求用户使用 CLI 或手动导出文件。

外部 agent 负责代码生成、数据分析、实验报告、图表生成和进一步推理。本产品只提供本地论文知识服务。

## 3. 用户模型

目标用户是个人研究者，包括学生、科研人员、独立研究者和工程型研究者。用户通常有若干研究 idea，每个 idea 下面有一组相关论文，希望在后续写代码、做实验、分析结果或写论文时，让外部 agent 能准确引用这些论文中的方法、结果和限制。

典型用户行为：

- 为一个新 idea 创建独立空间。
- 向空间中导入 PDF。
- 系统自动抽取论文的核心方法、结果、限制和可复用实验信息。
- 在外部 agent 中分析问题时，让 agent 调用当前 idea space 的文献知识。

## 4. 核心概念

### 4.1 Idea Space

Idea Space 是产品的一级组织单位。每个 space 对应一个研究 idea、课题方向或待验证假设。

每个 space 拥有独立的：

- 论文文件。
- 元数据。
- 解析文本。
- 结构化抽取结果。
- 全文索引。
- 可选向量索引。
- 用户笔记和标签。
- agent 访问权限上下文。

默认情况下，agent 连接某个 space 后，只能访问该 space 中的论文知识。这样可以减少跨项目污染，也符合科研项目的工作方式。

### 4.2 Paper

Paper 是导入到某个 idea space 的论文对象。

基础字段：

- 标题。
- 作者。
- 年份。
- DOI、arXiv ID、PubMed ID 或其他外部标识。
- 期刊或会议。
- 摘要。
- PDF 本地路径。
- 引用信息。
- 用户标签。
- 与当前 idea 的关系。

论文与 idea 的关系可以包括：

- 支持。
- 反驳。
- 启发。
- baseline。
- 方法来源。
- 相关背景。
- 结果对比。
- 未分类。

### 4.3 Passage

Passage 是论文中可被引用的文本片段。它是检索和溯源的基本单位。

字段包括：

- paper id。
- 章节。
- 页码。
- 段落序号。
- 原文文本。
- 解析置信度。
- 所属类型，例如 abstract、introduction、method、result、discussion、limitation、appendix。

agent 获得的任何文献证据都应该能追溯到 passage。

### 4.4 Knowledge Card

Knowledge Card 是从论文中抽取出的结构化知识单元。它不是最终结论，而是供 agent 检索和组合的文献片段。

通用类型：

- Problem：论文研究的问题。
- Claim：论文提出的主张。
- Evidence：支持某个 claim 的证据。
- Method：方法、流程、算法、protocol 或实验设计。
- Object：研究对象，例如数据集、样本、材料、模型、系统或人群。
- Variable：变量、参数、条件或干预因素。
- Metric：评价指标、测量方式或统计标准。
- Result：主要结果。
- Failure Mode：失败模式、异常现象或负面结果。
- Interpretation：作者对结果的解释。
- Limitation：局限性、适用边界或未解决问题。
- Practical Tip：可复用的操作经验或实验建议。

每个 card 都必须保存来源 passage，避免脱离原文的孤立摘要。

## 5. 核心功能

### 5.1 创建和管理 Idea Space

用户可以在本地 UI 中创建、打开、重命名、归档和删除 idea space。

每个 space 有一个简短说明，用于描述该 idea 的研究目标、当前假设和用户关心的问题。这个说明会参与后续论文相关性判断和 agent 检索。

第一版只需要单机本地管理，不需要同步和多人协作。

### 5.2 导入论文

支持以下导入方式：

- 拖拽 PDF。
- 输入 DOI。
- 输入 arXiv 链接或编号。
- 输入 PubMed ID。
- 粘贴 BibTeX。

导入后系统执行：

- 保存原始 PDF。
- 尝试补全元数据。
- 提取正文文本。
- 识别标题、摘要、章节、参考文献和页码。
- 去重检测。
- 建立初始索引。

如果元数据解析失败，用户可以手动编辑。

### 5.3 论文解析

解析流程应优先可靠，不追求一次性完美。

第一阶段：

- 提取 PDF 文本。
- 按页、章节和段落切分。
- 标记 abstract、introduction、method、result、discussion、limitation、appendix 等粗粒度章节。
- 生成 passage。

第二阶段：

- 从 passage 中抽取 knowledge cards。
- 将 card 与原文 passage 绑定。
- 给每个 card 记录类型、摘要、来源和置信度。

系统应该允许用户查看和修正抽取结果。

### 5.4 本地检索

检索应采用全文检索为主、向量检索为辅的混合策略。

全文检索用于精确匹配：

- 方法名。
- 数据集名。
- 指标名。
- 药物、材料、基因、模型、算法等专有名词。
- 作者、年份、论文标题。
- 参数值和实验条件。

向量检索用于补充语义相近但没有同词命中的内容。

默认排序应综合：

- 全文匹配分数。
- 向量相似度。
- 是否命中标题、摘要、方法或结果章节。
- 是否来自 knowledge card。
- 与当前 idea space 描述的相关性。
- 用户手动标注权重。

返回结果必须包含来源信息，包括论文、章节、页码和原文片段。

### 5.5 结构化文献视图

用户可以在 UI 中从不同角度查看当前 idea space：

- 论文列表。
- 按标签或关系分类的论文。
- 与 idea 相关的核心 claim。
- 方法列表。
- 实验变量和条件。
- 评价指标。
- 主要结果。
- 失败模式和局限性。
- 用户笔记。

这些视图不是为了替代论文阅读，而是帮助用户和 agent 快速定位相关文献内容。

### 5.6 Agent 接入

系统提供一个本地 MCP Server，让外部 agent 可以访问当前 idea space 的论文知识。

第一版不提供 CLI，不要求用户手动导出文件。用户在 UI 中选择一个 space，并启用本地 agent 接入。

MCP 工具建议包括：

- `list_spaces`：列出本地 idea spaces。
- `get_active_space`：获取当前启用的 space。
- `list_papers`：列出当前 space 的论文。
- `search_literature`：混合检索论文 passage 和 knowledge cards。
- `get_paper_summary`：获取单篇论文的结构化摘要。
- `get_methods`：查找与问题相关的方法。
- `get_metrics`：查找与问题相关的评价指标。
- `get_limitations`：查找相关局限性和适用边界。
- `find_failure_modes`：查找类似失败模式或异常现象。
- `find_similar_results`：根据实验观察查找相似文献结果。
- `compare_with_literature`：返回与某个观察或结果相关的文献证据。
- `get_evidence_for_claim`：查找支持或反驳某个 claim 的证据。
- `get_citation`：返回论文引用信息。

这些工具不生成最终科研结论，只返回文献上下文、结构化证据和原文来源。

### 5.7 典型 Agent 使用场景

场景一：实验遇到问题。

用户在外部 agent 中描述问题，例如某个方法效果不稳定。agent 调用本系统查找当前 idea space 中相似失败模式、相关限制、作者解释和可尝试的参数范围。

系统返回：

- 相似失败模式。
- 相关论文。
- 原文证据。
- 可能涉及的方法或变量。
- 作者在论文中的解释。

场景二：分析实验结果。

用户让 agent 分析一组实验数据。agent 读取数据后，总结观察现象，再调用本系统查找类似结果和文献解释。

系统返回：

- 类似结果的论文。
- 相关实验设置。
- 文献中的解释。
- 与用户结果可比较的指标和条件。
- 适用边界和风险。

场景三：设计下一步实验。

agent 根据用户 idea 和当前结果调用本系统，查找可复用方法、baseline、变量、指标和局限性。

系统返回：

- 可参考的方法。
- 常见 baseline。
- 推荐比较指标。
- 需要控制的变量。
- 可能失败的条件。
- 对应出处。

## 6. 数据隔离和本地存储

所有数据默认存储在用户本地机器。

建议本地目录结构：

```txt
app-data/
  spaces/
    <space-id>/
      space.json
      papers/
      parsed/
      cards/
      indexes/
      notes/
```

每个 space 独立存储自己的论文、解析结果、索引和笔记。

系统层面可以有一个全局数据库记录 space 列表和基础状态，但检索时必须明确限定 active space，避免 agent 查询到不相关项目。

## 7. 可靠性原则

产品应优先追求可追溯和可修正。

关键原则：

- 所有 agent 可见内容都必须带来源。
- 用户能看到原文 passage。
- 用户能修正错误元数据和错误抽取结果。
- 系统不把 LLM 摘要当作不可质疑的事实。
- 结构化抽取失败时，全文检索仍应可用。
- 向量检索只作为辅助召回，不能替代精确检索。
- 每次 agent 查询都限定在明确的 idea space 内。

## 8. MVP 范围

第一版应该包含：

- 本地 UI。
- 创建和切换 idea space。
- 导入 PDF。
- 基础元数据编辑。
- PDF 文本解析和 passage 切分。
- SQLite FTS 全文检索。
- 可选本地向量检索。
- 论文结构化摘要。
- knowledge card 抽取。
- 用户手动修正 card。
- 本地 MCP Server。
- agent 在当前 space 中检索文献、方法、结果、局限性和失败模式。

第一版不做：

- 云同步。
- 多人协作。
- 手机端。
- 完整引用管理器替代 Zotero。
- 复杂文献推荐。
- 自动系统综述。
- 实验数据管理。
- 文件导出。
- CLI。

## 9. 初始技术方向

本节仅描述倾向，不作为最终实现计划。

本地简单可靠的技术组合：

- UI：本地 Web UI 或 Tauri 桌面应用。
- 后端：Python FastAPI 或同等本地服务。
- 数据库：SQLite。
- 全文检索：SQLite FTS5。
- 向量检索：sqlite-vec、LanceDB 或本地 Qdrant。
- PDF 解析：PyMuPDF 起步，后续可接 Docling、GROBID 或 marker。
- 元数据补全：Crossref、arXiv、PubMed、Semantic Scholar，可作为可选联网能力。
- Agent 接入：MCP Server。
- LLM 抽取：支持用户配置本地模型或外部 API。

如果第一版目标是最简单可靠，建议先使用 SQLite + FTS5 + PyMuPDF + MCP。向量检索和更复杂 PDF 解析可以作为第二阶段能力。

## 10. 成功标准

MVP 成功的判断标准：

- 用户可以为一个 idea 建立独立论文空间。
- 用户可以导入一组论文并完成基本解析。
- 用户可以搜索到论文中的关键方法、结果、局限性和失败模式。
- 外部 agent 可以通过本地接口查询当前 space 的文献知识。
- agent 获取的每条文献信息都能追溯到具体论文和原文片段。
- 当用户要求 agent 分析问题或实验结果时，agent 能从知识库中获取相关文献依据，而不是只依赖模型记忆。
